# 家庭基金量化系统 MVP 设计

日期：2026-06-10

## 1. 背景

本系统用于管理家庭内部理财基金。它不是公开销售的基金产品，也不是自动投顾系统，而是一个家庭资产运营工具：把家庭投入资金视为一只内部基金，通过份额、净值、现金和持仓账本来持续跟踪收益、仓位和风险。

## 2. 第一版目标

第一版先做轻量 Web 系统，部署到腾讯云，供家庭内部使用。

第一版必须完成：

1. 手动录入外部资金投入和赎回。
2. 手动录入内部产品买入、卖出和手续费。
3. 按真实持仓产品做 T+1 数据更新。
4. 每日生成家庭基金估值记录。
5. 展示总资产、现金、份额、单位净值、累计收益、昨日收益、持仓和仓位偏离。
6. 页面内展示基础预警。
7. 支持管理员编辑，家人只读查看。

第一版不做：

1. 自动下单。
2. 微信、短信、邮件或企业微信主动推送。
3. 券商、支付宝、天天基金账单导入。
4. 复杂买点或卖点策略。
5. 完整历史回测。
6. 多家庭、多组合、多租户 SaaS 能力。

## 3. 成功标准

第一版上线成功的硬指标：

1. 管理员能录入一笔外部资金投入，系统按当日单位净值生成家庭基金份额，并增加现金。
2. 管理员能录入一笔产品买入，系统减少现金、增加产品持仓，家庭基金总份额不变。
3. 系统在 T+1 数据更新后，能重新计算总资产、单位净值、昨日收益和各产品仓位。
4. 看板能显示最新数据日期、更新状态、持仓市值、盈亏、现金余额和仓位偏离。
5. 某个产品数据更新失败时，不覆盖旧价格，并能在页面显示失败状态。
6. 家人账号只能查看，不能修改流水、配置和数据。

## 4. 技术路线

采用方案：Python FastAPI + 数据库 + 轻量 Web 前端。

选择理由：

1. Python 适合金融数据抓取、净值计算、定时任务和后续回测。
2. FastAPI 结构简单，适合第一版快速上线。
3. 账本和估值规则可以通过后端测试保护。
4. 腾讯云部署成本低，后续易维护。

第一版正式部署使用 PostgreSQL。本地开发可以临时使用 SQLite 做快速验证，但生产环境以 PostgreSQL 为准。数据库结构必须可迁移、可备份。

## 5. 核心账本模型

系统账本分为两层：家庭基金份额账本和内部持仓交易账本。

### 5.1 家庭基金份额账本

这层只处理外部资金变化。

支持流水类型：

1. `SUBSCRIBE`：外部资金投入。
2. `REDEEM`：外部资金赎回。
3. `DIVIDEND_IN`：现金分红进入基金账户。第一版保留类型，暂不做复杂分红处理。

规则：

1. 第一笔投入前，总份额为 `0`，展示单位净值为 `1.0000`。
2. 第一笔投入按 `1.0000` 生成份额。
3. 后续投入生成份额：`新增份额 = 投入净金额 / 当日单位净值`。
4. 赎回减少份额：`减少份额 = 赎回净金额 / 当日单位净值`。
5. 外部投入和赎回不计入投资收益。
6. 家庭基金单位净值只反映内部持仓涨跌、现金、费用和已实现盈亏。

### 5.2 内部持仓交易账本

这层只处理家庭基金内部如何投资。

支持流水类型：

1. `BUY`：买入基金、ETF 或其他产品。
2. `SELL`：卖出基金、ETF 或其他产品。
3. `FEE`：手续费或交易费用。
4. `CASH_ADJUSTMENT`：现金人工修正，必须填写备注。

规则：

1. 买入减少现金，增加产品持仓。
2. 卖出增加现金，减少产品持仓。
3. 内部买卖不改变家庭基金总份额。
4. 每个产品记录持有数量、成本、当前价格、市值、浮动盈亏和已实现盈亏。
5. 现金是显式资产，不用倒推方式隐藏处理。

## 6. 每日估值

系统每天基于现金、持仓和最新可用价格生成每日估值记录。

每日估值包含：

1. 日期。
2. 总资产。
3. 现金余额。
4. 总份额。
5. 单位净值。
6. 累计净值。
7. 当日收益。
8. 各产品市值和仓位快照。

单位净值计算：

```text
单位净值 = 总资产 / 总份额
```

如果总份额为 `0`，展示单位净值为 `1.0000`，不生成投资收益。

总资产计算：

```text
总资产 = 现金余额 + 所有持仓产品市值
```

## 7. 产品与数据更新

产品配置和真实持仓分开处理。

产品配置保存目标资产类别、目标仓位、候选场内代码和候选场外代码。真实持仓只来自管理员录入的实际买入产品。

数据更新规则：

1. 只更新真实持仓产品。
2. 按真实买入渠道适配数据源。
3. 场内 ETF 使用场内行情或基金相关数据。
4. 场外基金使用基金净值和净值日期。
5. 第一版只承诺 T+1，不承诺盘中实时。
6. 某产品未披露新数据时，保留上一可用数据，并标记数据日期。
7. 某产品更新失败时，不覆盖旧数据，只记录失败原因。
8. 各产品独立更新，单个产品失败不影响其他产品。

每条价格数据保存：

1. 产品代码。
2. 价格或净值。
3. 数据日期。
4. 数据来源。
5. 拉取时间。
6. 成功或失败状态。
7. 失败原因。

更新方式：

1. 服务器每天定时自动更新。
2. 管理员可以手动补跑昨日数据。
3. 管理员可以手动补跑指定日期。

## 8. Web 系统结构

后端模块：

1. 账本模块：处理资金投入、赎回、内部买卖、手续费、现金和份额计算。
2. 估值模块：生成每日估值。
3. 数据更新模块：拉取真实持仓产品 T+1 数据，记录状态。
4. 产品配置模块：维护资产类别、目标仓位、候选产品和真实持仓产品。
5. 预警模块：第一版只做页面内提示。
6. 权限模块：管理员可编辑，家人只读。

第一版页面：

1. 总览看板：总资产、单位净值、累计收益、昨日收益、现金余额、最新数据日期、当前预警数量。
2. 持仓页面：产品名称、代码、资产类别、持有数量、成本、最新价格、市值、盈亏、当前仓位、目标仓位、仓位偏离。
3. 流水录入页面：外部投入、赎回、内部买入、卖出、手续费和备注。
4. 数据更新页面：自动更新时间、各产品数据日期、成功或失败状态、手动补跑按钮。
5. 预警页面：仓位偏离、数据更新失败、产品长时间未更新和现金仓位异常。

## 9. 权限模型

第一版只支持家庭内部简单权限。

角色：

1. 管理员：录入流水、修改配置、触发数据更新、查看全部数据。
2. 查看者：查看看板、持仓、净值和预警，不能修改数据。

不做：

1. 多家庭隔离。
2. 复杂权限矩阵。
3. 对外注册。

## 10. 第一版预警

第一版预警只做确定性提示，不做投资建议式判断。

支持三类预警：

1. 仓位偏离预警：当前仓位偏离目标仓位超过配置阈值。
2. 数据更新预警：真实持仓产品更新失败，或连续多个交易日没有新数据。
3. 现金仓位预警：现金过高或过低。

第一版不做：

1. 复杂买点或卖点策略。
2. 技术指标组合信号。
3. 自动交易。
4. 收益承诺式提示。

## 11. 后续扩展方向

第一版完成后，再分阶段扩展：

1. 观察仓位配置。
2. 买点和卖点规则。
3. 历史数据回测。
4. 定期再平衡回测。
5. 仓位偏离再平衡回测。
6. 页面内预警稳定后，接入微信、企业微信或邮件推送。
7. 账单导入。
8. 数据库和部署结构升级。

回测模块后续复用：

1. 产品历史价格或净值。
2. 目标配置。
3. 调仓规则。
4. 观察仓位。
5. 预警规则。

## 12. 部署与维护

部署目标是腾讯云服务器。

第一版部署要求：

1. Web 系统可通过浏览器访问。
2. 手机和电脑均可查看。
3. 定时任务在服务器上每天执行。
4. 数据库可备份、可恢复。
5. 数据更新日志可追溯。
6. 关键计算规则有测试保护。
7. 部署步骤写入文档。

维护原则：

1. 账本流水不可随意覆盖。
2. 人工修正必须保留备注。
3. 数据更新失败必须记录原因。
4. 估值结果必须可从流水、持仓和价格数据复算。
5. 后续云服务器权限交接后，由 Codex 参与部署、更新、排障和功能迭代。

## 13. 风险边界

本系统是家庭理财运营工具，不是自动投顾系统。

系统可以帮助：

1. 计算家庭基金净值。
2. 追踪持仓和仓位偏离。
3. 提醒数据异常和配置偏离。
4. 后续辅助做回测和规则提示。

系统第一版不会：

1. 自动下单。
2. 保证收益。
3. 替代人工投资决策。
4. 对外提供投资建议。
